Section 01

The Chef & the Kitchen

Before we define an AI agent technically, here is an analogy that maps perfectly to every trait. A chef given a goal — cook the perfect dish — exhibits every behaviour we expect of an intelligent agent. Study this picture once and you will have an intuition that no textbook definition can give you. Click any trait node in the map to jump to its detailed card.

The Setup

A chef walks into a kitchen. His goal is to cook a specific dish. He looks around — he sees a blender, a grinder, a fridge, and a gas oven. Those are his tools. He recalls the recipe (his plan), picks up the right tool for each step, executes the steps in sequence, tastes as he goes, adjusts if something is off, and stops when the dish is ready. No one tells him what to do next. He figures it out.

That entire loop — perceive → plan → act → observe → adjust → stop — is exactly what an AI agent does.

Goal-Directedness

The agent operates in service of an explicit end-state. All its actions are evaluated against whether they bring it closer to that goal. It doesn't act randomly; every step is purposeful.

Chef Everything the chef does — choosing tools, tasting, adjusting — is aimed at one thing: the dish. Not cleaning the kitchen, not restocking the fridge.
Agent Given "book me the cheapest flight to Mumbai next Tuesday," every web search, every comparison, every filter is aimed at that goal. It does not browse randomly.
Test question: If you remove the stated goal, does the system still know what to do? If not — it is goal-directed. A goalless agent is just a script.

Perception

The agent senses and reads its environment before and during action. It does not act blind. Perception provides the raw material for reasoning.

Chef Before picking up a single utensil, the chef surveys the kitchen — what tools are available, what ingredients are in the fridge, what is the stove's current temperature.
Agent An AI coding agent reads the repository structure, open issues, error logs, and test results before writing a single line of code. That reading is perception.
Test question: Does the system read the current state of its environment, or does it just execute a fixed sequence regardless of context?

Planning

Before acting, the agent constructs a sequence of steps to reach the goal. It reasons about what needs to happen, in what order, and why. Planning is pre-action reasoning.

Chef The chef decides the recipe — the full sequence of steps — before touching a single ingredient. He considers timing: marination before cooking, sauce before plating.
Agent A research agent, asked to write a market report, plans: first search for data, then find recent news, then synthesise, then write — in that order, not randomly.
Test question: Does the system decide how to approach a task before executing, or does it just execute the first action that comes to mind?

Memory

The agent retains information across steps within a task. It does not restart from scratch at each action. Memory allows context to accumulate and inform later decisions.

Chef The chef remembers that he salted the sauce three minutes ago, so he does not salt it again. He holds the recipe in mind across the entire multi-hour session.
Agent An agent troubleshooting a bug remembers which fixes have already been tried, so it does not suggest the same solution twice. The conversation history is its working memory.
Test question: If you ask the system to do Step 5, does it know what happened in Steps 1–4, or does each step feel like a fresh start?

Tool Selection & Use

The agent identifies which external capability is required for a given step, invokes it, and uses its output. It does not do everything internally — it extends its capability through tools.

Chef For the coconut paste, the chef uses the blender — not his hands. For spices, the grinder. He selects the right tool for the right step, not the most familiar one for everything.
Agent For current stock prices, an agent calls a finance API. For web data, it calls a search tool. For code execution, it calls a REPL. It knows what it cannot do natively.
Test question: Does the system decide which tool to use and when, or is the tool use hard-coded in advance ("always search, then always summarise")?

Sequential Action-Taking

The agent executes a series of actions over time. Each action builds on the previous one. The output of one step becomes the input for the next. This is not a single-shot response.

Chef Chop → Grind spices → Heat oil → Sauté → Add ingredients → Simmer → Taste → Plate. Each step depends on the previous one. You cannot plate before you cook.
Agent Search → Scrape results → Summarise → Cross-check facts → Draft report. The draft uses the summary which used the scrape. Remove any step and the chain breaks.
Test question: Does the system take multiple actions over time where each depends on the previous, or does it produce a single output and stop?

Autonomy

The agent decides what to do next without a human providing the prompt for each step. Once the goal is given, it self-directs the execution. The human is the what; the agent owns the how and when.

Chef Nobody tells the chef "now add salt" or "now turn down the heat." He decides those things himself, in real time, based on what he perceives and what he knows.
Agent An agent asked to "refactor this codebase" decides which files to read first, which functions to touch, and in what order — without a human approving each micro-decision.
Test question: After the first instruction, does the human need to keep prompting for each step? If yes — it is not autonomous. It is a chatbot with extra steps.

Observe–Reason–Act Loop

The agent checks the outcome of each action, reasons about whether it is on track, and adjusts. This closed-loop behaviour is what separates an agent from a script. Scripts execute. Agents adapt.

Chef The chef tastes the curry. Too bland — he adds more spice. Too thick — he adds water. He does not just execute the recipe blindly; he course-corrects based on real-time feedback.
Agent A coding agent runs the tests after each change. If tests fail, it reads the error, reasons about the cause, and tries a different fix. Observe → reason → act → repeat.
Test question: If an action produces an unexpected or wrong result, does the system notice and try again differently — or does it proceed regardless?

Trait 09
EnhancingEnhancing traits increase capability and reliability, but their absence does not automatically disqualify a system.
Termination Criterion

The agent knows when the task is done. It has a stopping condition — either the goal state is reached, a maximum attempt limit is hit, or a human signals completion. Without a termination criterion, an agent loops forever.

Chef The chef tastes the dish and knows it is ready. He does not keep cooking indefinitely "just in case." The goal state (ready to serve) is the stopping condition.
Agent An agent stops when: the task succeeds, a maximum retry count is exceeded, a human intervenes, or a confidence threshold is met. A well-designed agent never loops infinitely.
Test question: Under what condition does the system stop? If the answer is unclear, the agent is poorly designed — and potentially unsafe.

The 5-Question Classifier

Think of any process — a software system, a business workflow, an AI product — and answer these five questions. The verdict updates in real time as you answer, and shows exactly which traits are missing.

Answer all five questions about the process you are evaluating. Base your answers on what the system does, not what a human does around it.

0/5
questions answered yes — verdict updates automatically
Q1
Does the system decide what action to take next on its own, without a human prompt for each step? i.e., is the "what to do next" decision made by the system, not by a person clicking a button?
Q2
Does the system use tools or external capabilities dynamically, choosing which tool to invoke based on context? e.g., decides to search the web, call an API, read a file — based on need, not a fixed sequence.
Q3
Does the system observe the result of its actions and adjust subsequent steps accordingly? i.e., if a step fails or produces unexpected output, does the system notice and try a different approach?
Q4
Does the system operate across multiple steps over time, where each step depends on the output of the previous? i.e., is this a multi-action trajectory rather than a single-shot response?
Q5
Does the system have a defined stopping condition it checks itself — rather than stopping only because a human ends the session? e.g., stops when the goal is met, max retries reached, or a success criterion is satisfied.
Classification
Workflow / Pipeline
This process is not agentic. It is a deterministic workflow or pipeline — a sequence of pre-defined steps that execute in a fixed order, where the path is designed by humans in advance. The system does not decide, adapt, or self-direct. It executes.

This is not a criticism — workflows are often the right choice. They are predictable, auditable, and safe. Consider an agentic approach only if the task is too dynamic or complex for a fixed path.
Classification
Borderline / Assisted
This process has some agentic properties but not all. It may have dynamic tool use or multi-step execution, but lacks full autonomy, a feedback loop, or a self-assessed stopping condition. It is an assisted system — more capable than a pure workflow but still dependent on human direction at key decision points.

A useful heuristic: if a human must approve or prompt at more than 30% of the steps, it leans workflow. If the system runs for extended periods without human input, it leans agentic.
Classification
Fully Agentic
This process is genuinely agentic. The system exhibits autonomous decision-making, dynamic tool selection, a self-correcting feedback loop, multi-step execution, and a self-assessed stopping condition. It is not just executing instructions — it is pursuing a goal.

With agentic capability comes responsibility: ensure the system has guardrails, human oversight at critical junctures, clear scope boundaries, and a reliable way to intervene if something goes wrong.

The Decision Framework

Five diagnostic lenses. For each lens, "workflow answer" and "agentic answer" are given. The more your system's behaviour matches the agentic answer, the further right it sits on the spectrum.

Is the execution path known in advance?
WF Yes — a flowchart could fully describe it
AG No — the path emerges from the goal and environment
What happens when something unexpected occurs?
WF Fails, alerts a human, or goes to a fixed error path
AG Adapts — retries, reconsiders, tries an alternative
How does the system use tools?
WF Tool use is hard-coded: always step A then tool B
AG Selects tools dynamically based on task requirements

WF Baked into the design — rules encode human decisions

CHOOSE WORKFLOW WHEN
  • The path is fully known and repeatable
  • Auditability and predictability are critical
  • Failure modes must be pre-defined
  • Regulatory compliance requires determinism
  • Speed and cost matter more than flexibility
CHOOSE AGENTIC WHEN
  • The task is too dynamic for a fixed path
  • Conditions change mid-execution
  • Multiple tools must be combined flexibly
  • The human should not be in the loop per step

An agent is built around an LLM. The LLM is the brain; the agent is the whole person — with hands (tool calls), memory (context/storage), eyes (perception), and a goal. You make an agent by wrapping an LLM in an orchestration layer that gives it those capabilities.

However, ChatGPT with tools enabled (web search, code execution, file reading) begins to exhibit agentic properties within a single conversation. It can search, read the result, reason, search again, and synthesise. That is a feedback loop with tool use — borderline to agentic depending on how autonomous the step-to-step decision-making is.

The key test: Is the system taking actions in the world across multiple steps, with the output of one informing the next, without human prompting in between? If yes — agentic. If it is still one prompt, one reply — not agentic.
This is the most important distinction in practice. The answer turns on three things:
  • Decision-making: In a workflow, transitions between steps are determined by pre-written rules or human action. In an agentic pipeline, the system decides the next step based on what it observes.
  • Path predictability: A workflow's execution path can be fully diagrammed in advance. An agentic pipeline's path emerges at runtime — it depends on what the agent encounters.
  • Adaptation: A workflow handles failures through pre-defined error handling. An agent reasons about the failure and tries a different approach.
A useful shorthand: workflows execute a plan; agents make a plan and execute it adaptively.

The chef analogy makes this vivid: a workflow chef follows a recipe rigidly — if an ingredient is missing, the recipe fails. An agentic chef surveys the kitchen, adapts the recipe to what is available, and still delivers the dish.
Yes — the tool set being pre-defined does not compromise autonomy. The chef's tools are also pre-defined (the kitchen has what it has). Autonomy is not about having infinite tools; it is about deciding which tool to use and when.

The relevant question is: Is the choice of which tool to invoke made by the system at runtime, based on context? If yes — that is tool autonomy. If the sequence is hard-coded ("always call tool A, then tool B, then tool C") — that is a workflow with tools, not an agent.

Autonomy lives in the decision layer, not in the size or openness of the tool set. A highly constrained agent (two tools, narrow domain) can still be genuinely autonomous within those constraints.
A multi-step chatbot has a conversation across multiple turns. Each turn requires a human to provide the next input. The chatbot responds; the human responds; the chatbot responds again. The human is in the loop at every step.

An agent runs autonomously across multiple steps without the human prompting each one. After the initial goal is given, the agent plans, acts, observes results, and iterates — all without waiting for human input between steps.

The diagnostic question: Between Step 1 and Step 5, how many times did a human have to press a button or type something? If the answer is 4 — it is a chatbot. If the answer is 0 — it is an agent.

Example: a loan origination system might have a rigid workflow for document collection and compliance checks (deterministic, auditable), but an agentic component for underwriting analysis (adaptive, multi-source reasoning). The overall process is a workflow; the underwriting module is an agent.

No — classical RPA is not agentic. RPA bots are sophisticated workflow executors: they follow explicit, rule-based scripts to interact with UIs and systems. They use "tools" (screen scraping, form filling) but the sequence is entirely pre-defined.

The key deficiencies vs a true agent:
  • No adaptive reasoning: if the UI changes slightly, the RPA breaks. An agent would observe the change and adapt.
  • No dynamic tool selection: the script defines exactly which UI elements to click, in what order.
  • No feedback loop: if step 7 fails, the bot alerts a human — it does not reason about why and try differently.
However, "AI-augmented RPA" (RPA + LLM-based decision-making) begins to cross into borderline territory, where the AI component makes dynamic decisions while RPA executes them.
No — but three traits are non-negotiable. The minimum viable agentic system requires:
  • Autonomy: the system must self-direct at least some steps without human prompting.
  • Feedback loop: the system must observe the result of actions and adapt. Without this, it is a pipeline.
  • Goal-directedness: without a goal, the system has no basis for any decision.
The other six traits (perception, planning, memory, tool use, goal decomposition, termination) are present in well-designed agents and increase capability and reliability — but their absence does not automatically disqualify a system from being agentic.

Think of it as a spectrum score: 3 core traits = minimally agentic. All 9 traits = fully agentic. Most real systems score 4–7.
The risks are practical and significant:
  • Misaligned expectations: if a team builds what they call an "AI agent" that is actually a workflow, they will be disappointed when it cannot handle edge cases, and confused about why it fails in dynamic conditions.
  • Wrong architecture: workflows and agents require different engineering approaches. Building an agent architecture for a workflow problem wastes complexity; building a workflow for a problem that needs an agent creates brittleness.
  • Governance failures: truly agentic systems require different oversight — human-in-the-loop design, audit trails for decisions, scope constraints. Mislabelling a real agent as a workflow can mean insufficient controls are put in place.
  • Vendor confusion: many vendors label fixed pipelines as "AI agents" for marketing purposes. Knowing the distinction helps you evaluate claims accurately.
The precise use of language here is not academic — it has direct consequences for system design, risk management, and business outcome.
Use this readiness checklist:
  • Task clarity: Can you articulate the goal in one sentence? Agents need clear goals. Vague goals produce uncontrolled agents.
  • Tool availability: Are the tools the agent needs available as callable APIs or services? Agents without tools are just chatbots.
  • Failure tolerance: Can your organisation tolerate occasional wrong actions that need human correction? Agents are not perfect. Workflows are more predictable.
  • Oversight design: Do you have a plan for when to intervene? Human-in-the-loop checkpoints for high-stakes decisions are essential.
  • Evaluation framework: Can you measure whether the agent is achieving the goal? You cannot improve what you cannot measure.
If you can tick all five — you are ready for an agent deployment. If not — start with a well-designed workflow and add agentic capability incrementally.
